# 积分业务逻辑

本文面向产品、测试和开发人员，说明积分模块的账户模型、计算规则、状态流和异常边界。

## 核心概念

| 概念 | 说明 |
|------|------|
| 可用积分 | 用户当前可用于订单抵扣和积分兑换的积分 |
| 冻结积分 | 订单完成后处于售后保护期内、暂不可使用的奖励积分 |
| 欠账积分 | 退款回收已释放积分时，可用积分不足形成的待抵扣积分 |
| 积分流水 | 每一次积分变动记录，包含业务类型、方向、账户类型、变动前后值 |
| 奖励快照 | 订单完成时保存的赠送积分和订单项积分明细 |
| 抵扣记录 | 下单时保存的使用积分和抵扣金额 |
| 兑换单 | 用户在积分商城兑换商品后生成的履约单据 |

积分统一使用整数，不支持小数。

## 开关影响

| 开关 | 影响 |
|------|------|
| 积分总开关 | 关闭后客户端和商品编辑不展示积分入口，停止新发放、新抵扣、新兑换 |
| 积分赠送开关 | 关闭后订单完成不再生成新的赠送积分 |
| 积分抵扣开关 | 关闭后确认订单页不能使用积分抵扣 |

历史返还、回收和释放继续处理，避免旧订单和旧兑换单无法闭环。

## 积分账户

用户首次需要积分账户时，系统会自动创建账户。

账户聚合字段：

| 字段 | 含义 |
|------|------|
| `balance_points` | 可用积分 |
| `frozen_points` | 冻结积分 |
| `debt_points` | 欠账积分 |
| `total_income_points` | 累计获得积分 |
| `total_expense_points` | 累计扣减积分 |

规则：

- 可用积分增加时，如果存在欠账积分，会优先抵扣欠账。
- 可用积分扣减时，默认不允许扣成负数。
- 退款回收场景允许部分扣减，不足部分写入欠账积分。

## 消费返积分

订单完成后触发消费返积分。

计算来源优先级：

1. 商品配置为关闭、按金额或固定时，使用商品配置。
2. 商品配置为规格单独配置时，使用 SKU 配置。
3. SKU 继承或未单独配置时，使用全局消费返积分规则。

商品 / SKU 积分模式：

| 模式 | 含义 |
|------|------|
| `global` | 使用全局消费返积分规则 |
| `disabled` | 不赠送积分 |
| `ratio` | 每消费 1 元赠送指定积分 |
| `fixed` | 每件固定赠送指定积分 |
| `sku` | 商品按 SKU 配置计算 |
| `inherit` | SKU 继承商品或全局规则 |

计算示例：

| 场景 | 计算 |
|------|------|
| 全局每元 1 积分，订单项实付 128.50 元 | `128` 积分，按分转整数后向下取整 |
| 固定每件 10 积分，购买 3 件 | `30` 积分 |
| 单次上限 100 积分，理论奖励 168 积分 | 最终奖励 `100` 积分，并按订单项比例分摊 |

## 冻结与释放

奖励积分不会立即进入可用积分，而是先进入冻结积分。

原因：

- 订单完成后仍可能发生售后退款。
- 冻结期内退款可以优先回收冻结积分。
- 避免用户先使用奖励积分，再退款导致积分账户异常。

流程：

1. 订单完成。
2. 系统计算奖励积分。
3. 写入订单积分奖励快照和订单项快照。
4. 增加用户冻结积分。
5. 到达释放时间后执行释放任务。
6. 冻结积分减少，可用积分增加。
7. 如果用户有欠账积分，释放时优先抵扣欠账。

释放任务：

```bash
php think points:release
```

## 退款回收

退款会回收订单奖励积分，并按退款金额比例计算应回收积分。

回收顺序：

1. 优先回收订单仍冻结的积分。
2. 冻结积分不足时，从可用积分扣减。
3. 可用积分仍不足时，形成欠账积分。

示例：

- 订单奖励 100 积分，已释放。
- 用户当前可用积分只有 30。
- 退款需要回收 80。
- 系统扣减 30 可用积分，并形成 50 欠账积分。

后续用户获得积分时，会优先抵扣这 50 欠账积分。

## 积分抵扣

确认订单页会进行积分抵扣试算。

影响因素：

- 积分抵扣开关。
- 用户可用积分。
- 积分抵扣比例。
- 单笔订单抵扣上限比例。
- 用户本次请求使用的积分数量。

计算示例：

| 项 | 值 |
|----|---|
| 商品可抵扣金额 | 100.00 元 |
| 积分抵扣比例 | 100 积分抵 1 元 |
| 抵扣上限比例 | 50% |
| 用户可用积分 | 12000 |

结果：

- 最多可抵扣金额：`100.00 * 50% = 50.00`
- 最多可用积分：`50.00 * 100 = 5000`
- 如果用户请求使用 8000 积分，最终只使用 5000 积分。

下单成功后：

1. 扣减用户可用积分。
2. 写积分流水 `order_deduction`。
3. 写订单积分抵扣记录。

订单关闭时，未消费的抵扣积分会通过 `order_deduction_return` 返还。售后退款时，会按退款金额比例通过 `refund_deduction_return` 返还。

## 积分商城兑换

积分商城商品引用普通商品 SKU。兑换时会生成独立的积分兑换单，不走普通订单表。

兑换流程：

1. 校验积分总开关。
2. 校验用户已登录。
3. 校验积分商品启用、普通商品上架、SKU 启用。
4. 校验兑换数量在 1 到 99 之间。
5. 校验每人限兑数量。
6. 校验积分商品兑换库存和 SKU 库存。
7. 扣减积分商品兑换库存和 SKU 库存。
8. 扣减用户可用积分。
9. 保存兑换单商品、SKU、收货地址快照。
10. 写兑换单操作日志。

兑换接口支持 `idempotency_key`。同一用户重复提交相同幂等键时，返回已创建的兑换单，避免重复扣积分。

## 兑换单状态流

| 状态 | 值 | 来源 | 可流转到 |
|------|----|------|----------|
| 待发货 | `10` | 用户兑换成功 | 已发货、已关闭 |
| 已发货 | `20` | 后台发货 | 已完成 |
| 已完成 | `30` | 后台完成 | 终态 |
| 已关闭 | `90` | 用户取消或后台关闭待发货单 | 终态 |

关闭待发货兑换单时：

- 返还用户积分。
- 恢复积分商品兑换库存。
- 恢复普通 SKU 库存。
- 写兑换关闭返还流水。
- 写兑换单操作日志。

已发货和已完成兑换单不能通过关闭流程返还积分。

## 幂等规则

积分模块依赖业务类型和业务单号避免重复处理。

| 场景 | 幂等依据 |
|------|----------|
| 订单奖励 | 订单号 |
| 冻结积分释放 | 奖励记录和订单号 |
| 订单积分抵扣 | 订单号 |
| 兑换扣减 | 兑换单号 |
| 兑换返还 | 兑换单号 |
| 前台兑换提交 | 用户 ID + `idempotency_key` |

## 与其它模块的关系

| 模块 | 关系 |
|------|------|
| 商品 | 商品 / SKU 控制积分赠送模式，积分商品引用 SKU |
| 订单 | 确认订单试算积分抵扣和预计奖励，下单后扣减抵扣积分 |
| 订单状态机 | 订单完成后触发积分奖励和会员成长值 |
| 售后 | 退款回收赠送积分，并返还已抵扣积分 |
| 库存 | 积分兑换扣减 SKU 库存，关闭待发货兑换单恢复库存 |
| 客户端装修 | 积分总开关关闭时，个人中心积分入口会被过滤 |
