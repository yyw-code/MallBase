# 地区快照与失效治理

适用场景：
- 省市区街道四级联动
- 收货地址、运费模板、订单地址等引用行政区划的数据
- 地区主数据会更新，但业务历史数据必须稳定展示

## 核心规则

1. 业务表不能只存地区主表实时关联结果。
2. 业务表必须同时存：
   - 地区 `id`
   - 地区 `code`
   - 地区名称快照
   - 完整路径快照
3. 展示历史数据时优先使用快照，不直接依赖地区主表当前名称。
4. 地区主数据更新后，不自动批量改写业务数据。
5. 地区失效后，业务数据默认策略是：
   - 标记失效
   - 禁止继续使用
   - 必须重新编辑后恢复可用

## 设计要求

### 收货地址

- 至少保存：
  - `province_id/province_code/province_name`
  - `city_id/city_code/city_name`
  - `district_id/district_code/district_name`
  - `street_id/street_code/street_name`
  - `region_path_text`
- 地址详情展示使用快照字段。
- 下单、设默认、再次编辑前，要重新校验当前街道是否仍有效。
- 若地区失效：
  - `region_status = 0`
  - 写明 `region_invalid_reason`
  - 地址不可继续用于业务

### 运费模板

- 规则必须保存街道级：
  - `region_ids`
  - `region_codes`
  - `region_names`
  - `region_path_texts`
- 相同街道不能重复出现在多个规则中。
- 模板列表或详情需要能识别失效规则数量。
- 若规则命中街道失效：
  - 标记失效
  - 不自动迁移
  - 由人工重新选择有效街道

## 主数据维护约束

地区管理不是普通 CRUD，必须受控：

- 允许：
  - 查询
  - 新增子级
  - 改名称/排序/状态
- 高风险：
  - 删除
  - 改父级
  - 改 code
- 高风险操作必须先检查引用：
  - 收货地址
  - 运费模板
  - 其它依赖地区的业务表

## 实现建议

1. 地区联动统一走地区主表接口，不允许前端长期维护独立 JSON 作为运行时数据源。
2. 后端统一提供：
   - `children`
   - `path`
   - 合法性校验
3. 业务保存时由后端统一规范化地区快照，不能让前端自己拼路径文本。
4. 若地区库由外部 JSON 更新，更新后的重点是“重算业务有效性”，不是“自动覆盖业务数据”。
