# preview 演示站分支边界规则

## 适用范围

- 切换、创建、合并、提交、推送 `preview` 分支
- 从 `main` 合并到 `preview`
- 任何可能把 `preview` 改动带回 `main` 的操作
- 修改演示站专用能力，例如登录页演示账号、一键恢复演示数据、演示站广告、公开演示提示
- 调整演示站部署配置、容器名、域名、公开入口或数据恢复流程

## 分支职责

- `main`：主站与二开用户使用的干净主分支，不包含公开演示站专用入口或演示运营能力。
- `preview`：公开演示站部署分支，允许包含演示站专用功能。

## 强制规则

1. 业务通用功能先进入 `main`，再从 `main` 合并到 `preview`。
2. `preview` 专用功能只留在 `preview`，禁止反向合并回 `main`。
3. 禁止在 `main` 增加演示站专用入口、公开恢复按钮、演示账号选择、演示广告等能力。
4. 合并方向必须默认是 `main -> preview`；任何 `preview -> main` 操作都必须先向用户说明风险并获得明确确认。
5. 在 `main` 处理冲突时，不得为了省事接受 `preview` 侧演示站专用代码。
6. 在 `preview` 合并 `main` 后，需要确认演示站专用能力仍然存在且未被主分支覆盖。
7. 提交前必须检查当前分支和 diff，确认没有把演示站专用文件误带到 `main`。

## 操作流程

### 通用功能同步到演示站

1. 在 `main` 完成通用改动。
2. 提交 `main`。
3. 切到 `preview`。
4. 执行 `git merge main`。
5. 检查演示站专用功能仍可用。

### 演示站专用改动

1. 确认当前分支是 `preview`。
2. 只修改演示站专用代码。
3. 提交到 `preview`。
4. 不合并回 `main`。

### 遇到反向合并请求

先判断 diff 是否包含演示站专用能力：

- 若包含：默认拒绝直接合并，建议改为 cherry-pick 通用提交或手工搬运通用部分。
- 若不包含：仍需展示 Git 计划并获得用户确认后再操作。

## 自检清单

- [ ] 当前分支是否符合改动性质：通用改动在 `main`，演示站专用改动在 `preview`。
- [ ] 是否存在 `preview -> main` 反向合并风险。
- [ ] `main` diff 中是否误包含登录页演示账号、一键恢复演示数据、演示站广告等专用功能。
- [ ] `preview` 合并 `main` 后，演示站登录页和恢复入口是否仍保留。
- [ ] Git 计划是否明确写出合并方向、涉及文件、是否排除演示站专用改动。
